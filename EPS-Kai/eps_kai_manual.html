EPS-Kai Documentation

EPS-Kai（イージー・プロンプト・セレクター・改）は、YAMLに登録された構文やマクロ構造を整理し、Stable Diffusion用プロンプトの組み立て・生成環境そのものを構築していくための強力なアドオンシステムです。

本ドキュメントは、「操作マニュアル」「匠エディタ解説」「エンジニア向けの技術仕様書」の3つの階層で構成されています。

最初は “ボタンをぽちぽち押す” だけで大丈夫です。しかし触っていくうちに、「ここを自分好みに増やしたい」「独自のランダム（RM）を仕込みたい」という制作欲求へ、驚くほど自然にステップアップできる設計になっています。

0. YAMLの基本構築（超基礎）

本システムは 「YAML（ヤムル）」 という非常に単純なテキストファイルに沿ってすべてのボタンを表示しています。「YAMLなんて聞いたこともない！」という場合でも、以下のルールさえ理解すれば一瞬で自分用のボタンを作成できます。

① ドノーマルな登録（1行に1タグ）

もっとも単純な書き方です。左側に「ボタンに表示させたい日本語（名前）」、右側にコロン : とダブルクォート "" で挟んだ「実際の英語プロンプト（呪文）」を書きます。

正面: "straight"
笑顔: "smile"
コロン（ : ）の後ろには、必ず「半角スペース」を1個空けるという鉄の約束事があります。これを忘れるとエラーになってしまいます。

② カテゴリ（アコーディオン）による階層化

アコーディオンフォルダのようにすっきり「たたんで整理」するために、以下のようにインデント（行頭の半角スペース2文字）で親子構造を作ります。

アングル:
  正面: "straight"
  背後: "back view"

これで画面上には「アングル」という名のオレンジ色のカテゴリフォルダが作られ、中に「正面」「背後」のボタンが綺麗に格納されます。

  • さらに深く分ける（多層構造）：
    スペース4つ、6つと増やすことで、何階層でもフォルダの中にフォルダを作ることができます。＞多層構築の詳細なルールへ飛ぶ
  • フラグ（C/S/O/E）や色を付ける：
    ボタンに衣服 [O] などのアイコンを付けたり、ボタンの横にカラーパレットを表示する高度な表現も、1行の書き方次第で自由自在に拡張できます。＞フラグ・カラー指定の詳細解説へ飛ぶ

1. 導入コントロールパネル (5大ボタン)

EPS-Kaiをシステムに導入すると、WebUI上に生成環境を管理するための5つのコアコントロールボタンが出現します。（※おまけ機能のラジオ展開ボタン「📻」については後述します）

❇️展開

セレクター開閉

EPS-Kai のメインパネルを展開・格納します。

🖍️編集

匠エディタ起動

ビジュアル編集モード・匠エディタを全画面で起動します。

🔁更新

キャッシュ同期

YAMLや画像キャッシュを同期。Shift+クリックでキャッシュを全初期化して完全再スキャンします。

保存

キャッシュバックアップ

画像存在キャッシュ情報をバックアップファイル（JSON）としてPCへ保存します。

読込

キャッシュ復元

保存したバックアップファイルを読み込んで画像ステータス情報をその場で復元します。

2. 左右クリックとマクロ (NM/RM) の全挙動

EPS-Kaiの特徴は、ボタンが単なる「ON / OFF」ではなく、「どのボタンを、左右どちらのクリックで登録したか」によって挙動がスマートに切り替わる点にあります。以下がその全パターンの網羅です。

① 単一ボタンのクリック挙動

左クリック (実構文入力)

登録された「英語のプロンプトテキスト」をそのままテキストエリアへ流し込みます。

入力例: 1girl, blue eyes, smile

右クリック (マクロ参照入力)

プロンプトへは実構文ではなく、日本語の「ボタン名」を参照するマクロ構文を送信します。後からYAMLの実構文を変更しても生成情報が崩れません。

入力例: %%青目%%, %%笑顔%%

② フォルダ/カテゴリボタン of クリック挙動 (NM / RM)

複数のボタンを束ねている「カテゴリの親ボタン」をクリックした場合、強力な一括流し込み（NM）やランダム抽選（RM）が発動します。

カテゴリを 左クリック (NM: Normal Macro)

そのカテゴリ配下にあるすべてのプロンプトを一斉に展開して流し込みます。キャラクターの衣装セット全部位などを一撃で入力する際に使います。

入力例: red shirt, blue skirt, black socks, brown shoes

カテゴリを 右クリック (RM: Random Macro)

カテゴリ配下の候補から「生成のたびに完全にランダムで1つだけ」を抽選するランダムマクロを構築します。

入力例: %%@表情:笑顔系@%%
💡 RM（ランダム）の別の書き方
カテゴリの右クリックだけでなく、単一ボタンのプロンプト登録枠に直接 %% 笑顔 ;; 怒り ;; 悲しみ %% のように ダブルコロン ( ;; ) で区切ってテキストを記述しても、自動的にランダム抽選（RM）として機能します。（大外を%２つづつで囲う必要があります）

3. プロンプト登録状態と色ルール

ボタンが今「どこに」「どうやって」登録されているかは、ボタンのグラデーション色だけで瞬時に判断できるようデザインされています。

ボタン外観現在の登録ステータス解説
◇ 青目ポジティブ：左登録（実構文）ポジティブプロンプトへ通常登録されています。
◇ 青目ポジティブ：右登録（マクロ）ポジティブプロンプトへマクロ（%%形式%%）で登録されています。
◇ 悪い手ネガティブ：左登録（実構文）ネガティブプロンプトへ直接実構文が流し込まれています。
◇ 悪い手ネガティブ：右登録（マクロ）ネガティブプロンプトへマクロ形式で登録されています。
◇ 青目ポジティブ：両登録（左右）左登録と右登録の両方が、同じプロンプト内で重複登録されています。
◇ 悪い手ネガティブ：両登録（左右）ネガティブ側で左右両方が同時に登録されています。
◇ 矛盾ポジネガ重複警告（コンフリクト）同じタグがポジティブとネガティブの両方に登録されている場合に発動する、紫のストライプ警告です。押し間違いを防ぎます。

EPS-Kaiのサムネイルおよびボタンマーク（◇や★など）の表示は、「フラグの有無」と「実在する画像ファイル」の組み合わせによって完璧に制御されています。
以下の表は、その全ての表示パターンを網羅したものです。

表の下半分（フラグなしボタン）に注目してください。フラグがなくても、画像を用意しさえすれば「★」や「◉」が付き、サムネイルは完璧に動作します。

6. 検索機能・+Prompt・一括抽出・Fav（お気に入り）機能

ボタンが増えるほど威力を発揮するのが、強力なフロントエンド検索と履歴機能、およびヘビロテボタンを一瞬で呼び出せるお気に入りシステムです。

🔍 検索窓と「+Prompt」チェックボックス

検索窓の横にある +Prompt チェックボックスのON/OFFで、検索対象をスマートに切り替えることができます。

  • OFF (デフォルト): 「ボタン名（例：笑顔）」のみを検索対象にします。UI上のボタンを探したい時にノイズなくスッキリと検索できます。
  • ON (チェックあり): ボタン名に加えて、内部に登録されている「英語のプロンプト構文（例：smile）」も検索対象に含めます。英語のタグ名しか思い出せない時に非常に便利です。

🎯 100%漏れなき「一括抽出マクロ」と表示上限

  • 検索結果表示の上限設定: 検索結果のボタン表示件数は、ブラウザレンダリングの過負荷（プチフリーズ）を防ぐため最大100件までに制限（カットオフ）されています。
  • 一括マクロは100%送信: 表示が100件に制限されている場合でも、検索窓の下に出現する「検索結果（◯件）」の統合マクロボタンをクリックして送信されるマクロ（%%SEARCH:ワード%%）は、非表示になっているものも含めた「条件にマッチした全ボタン内容（100%）」をサーバー側ですべて漏れなく展開・流し込みを行います。安心して一括抽出をご利用ください。

💖 ヘビロテボタンを最速展開する「Fav（お気に入り）機能」

頻繁に使用するお気に入りのプロンプトやマクロボタンを、カテゴリやタブの垣根を越えて一箇所にまとめることができる機能です。

  • 最上位への常駐： お気に入りに登録されたボタンは、メインパネル最上部に新設される専用の「Favエリア」に自動的に集約・常駐します。
  • お気に入り編集モードの切り替え： 「Fav:」ラベルの横にある「編集」ボタンをクリックすると、ボタンが黄色く明るく光る「Fav編集モード」に移行します。
    • 追加する時： 編集モード中に、メインのカテゴリーからピン留めしたい任意のボタンを左クリックします。これだけでFav領域にピカッと追加されます。
    • 削除する時： 編集モード中に、Favエリア内にある不要なボタンを左クリックすると、即座に一覧から除外されます。
    整理が終われば、もう一度「終了」ボタンを押して通常モードに戻します。
  • お気に入りボタンの通常稼働： Favエリアに常駐した一軍ボタンは、通常時以下のように反応します。
    左クリック

    Favエリアからそのまま実構文（例：1girl）をプロンプトへ流し込みます。

    右クリック

    Favエリアからお気に入りのマクロ参照（例：%%1girl%%）をプロンプトへ流し込みます。これにより実構文を後から書き換えても安全な統合制御を維持します。


  • 匠エディタへの特急取り込み（B系列）： 本家EPSで登録したFavタグ群は、匠エディタの「B系列（例：favoriteB00）」を使用することで、ワンクリックでYAMLファイル内に新しいセットとしてそっくりそのまま流し込み・保存（特急登録）することが可能です。詳しくは匠エディタ解説 10. 特急取り込みシステムをご覧ください。

7. EPS-Kai の世界構築

このツールは「あらかじめ用意されたデータを使う」だけのものではありません。髪型、表情、ポーズ、衣服、ライティング、お気に入りの画質設定、キャラクターLoRA……あなただけの「プロンプト設計図」をYAMLとして構築し、それを取り込むことで、Stable Diffusionをあなた専用の最強のAIスタジオへとカスタマイズしていく、そのために存在しています。

匠エディタ 1. エディタ概要

YAMLデータを直接テキストエディタで書き換えるのは、インデントのズレ（YAML構文エラー）や、タイポ、上書きミスの原因になります。「匠エディタ」は、WebUI内部で稼働する、YAMLおよびサムネイル画像のビジュアル・オーサリング環境です。

「EPS-Kai本体」がプロンプトを使用するUIであるなら、「匠エディタ」はあなたの生成環境を強力に**育成・拡張**するためのエディタです。

匠エディタ 2. 開始前の大前提：ファイルの物理準備

🚨 【初見殺しを回避するための超重要事項】
匠エディタ上で新規にお気に入りパレットを編集・構築、あるいは本家EPSから特急取り込みを行うためには、最初から対象のYAMLファイル（物理ファイル）がディスク上に用意されているか、後述する「📄新規作成」ボタンからファイルを作成しておく必要があります。

⚠️ お気に入り用ファイル名「favorite」の絶対命名ルール

匠エディタおよびEPS-Kai本体でお気に入り機能（A系列、B系列等）を使用するための物理YAMLファイルは、必ず以下の厳格なフォーマットに従って命名してください。

【命名形式】 favorite[英字1文字][数字2桁].yml （例）favoriteA00.yml, favoriteA01.yml, favoriteB00.yml, favoriteB05.yml

※【警告】システムの読み込み判定・プルダウン抽出・保存プログラムは、この正規表現（英字1文字＋数字2桁）を厳しくチェックしています。 favA00.yml や favoriteA0.yml、あるいは日本語が含まれるファイル名といった**ルール違反のファイル名はシステムによって完全に無視（非表示・非接続）**されます。

◉ 処理の都合デフォルトでAとBに分けており、Bが特急専用フラグとしていますが、お気に入り登録については、A,C~Z＋00〜99のファイル名に対応しているので、自分なりのルールでお使いください。（Cは背景、Fはポーズ、＠15は特別なルール・・・・など）

匠エディタ 3. 基本画面構成

「🖍️編集」ボタンを押すと、画面全体を暗転させてクリエイターモードへ移行します。画面は上下に分割されており、効率的な編集作業を可能にしています。

上部：Viewerエリア

YAML内のカテゴリ（セクション）が美しく一覧表示されます。開閉（アコーディオン）や、ドラッグでの縦幅リサイズが可能です。

下部：Editorエリア

選択したアイテムの「名称(YAML KEY)」「フラグ(C/S/O/E)」「実行プロンプト」を直接書き換える、高機能な編集フォームです。

右側：Thumbnailエリア

サムネイル画像をドラッグ＆ドロップでドロップ可能。保存名がリアルタイムに表示され、画像のみの瞬時保存にも対応しています。

匠エディタ 4. 独立稼働: プロンプト編集機能

「ボタンの名前を変えたい」「内部のプロンプト（英語）を修正したい」「フラグ（C/S/O/E）を切り替えたい」というテキスト・設定ベース of 編集機能です。

  • ターゲットを選択し、フォームを書き換えます。
  • 画像（サムネイル）を用意していなくても、この機能単独で「💾 保存」が可能です。
  • 保存した瞬間にYAMLが書き換わり、プロンプトの出力内容が即座にアップデートされます。

匠エディタ 5. 独立稼働: 超高速サムネ投入機能

「テキストはそのままでいいから、サムネイル画像だけを追加・更新したい」という場合に特化した、ドラッグ＆ドロップ機能です。

  • 右側の円形ドロップゾーンへ、画像をポイッとドラッグ＆ドロップします。
  • どんなに大きな解像度の画像でも、ブラウザの機能を使って「最高品質（ランチョス風）」で 320x320 のPNGへとその場で自動圧縮されます。
  • 「📷 画像のみを保存」を押すことで、YAML（テキスト）には一切触れず、画像ファイルだけを瞬時に生成・上書きします。
  • 保存された瞬間にEPS-Kaiの画像キャッシュが上書きされ、Viewerエリアの ◇ マークがその場で一瞬で色付き of ◉ マークへと自動変身します！

匠エディタ 6. 鉄壁の保存前重複禁則ガード

匠エディタの安全設計は、入力フォームとセーブロジックの両方に強力なロックをかけてミスを完全に封殺します。

① 重複エラーバナー & 保存ボタン無効化

同じカテゴリ内にすでに存在する名前（KEY）を入力しようとすると、画面上部に警告バナーが表示され、保存ボタン（💾 保存）が自動的に無効化（グレーアウト・不透過度0.4）されます。

重複エラー：「にっこり [C]」は既に使用されています。

② 鉄壁のセーブプログラム防衛（禁則ガード）

万が一、無効状態のはずの保存ボタンが裏技的に呼び出されたとしても、プログラムの最前面において重複チェックを再検証し、セーブ動作そのものを瞬時に完全に遮断（禁則）します。これにより、意図しない不正上書きセーブやデータの破壊事故が完全にゼロになります。

匠エディタ 7. 画面右上グローバルアクション (大枠の操作)

右上の「匠」アイコンの真横に、ルート階層（一番外側のファイルレベル）の構造をダイナミックに変更するための3つのグローバルボタンが常駐しています。
これらはYAMLファイル全体の「皮むき」や「入れ子（ネスト）」を一瞬で実行する強力な機能です。

💡 初見殺し回避ポイント
これらのボタンは「現在選択しているYAMLファイル全体」の構造に影響を与えます。もし意図しない結果になった場合は焦らず、もう一度「外枠を消す」などで構造を戻すことができます。

📦 すべてを新枠で包む

現在のファイル内にあるすべてのカテゴリを、新しい1つの大きなフォルダ（親カテゴリ）の中に移動させます。 ファイル内がごちゃごちゃしてきた時に、全体を一段深く（インデントを下げて）まとめたい場合に最適です。

※展開先に同じ名前のカテゴリが既に存在する場合は、上書き破壊を防ぐための安全ガードが作動し、処理が中断されます。

# ▼ 実行前 外見: 髪型: ショート: "short hair" 服装: シャツ: "shirt" # ▼ 実行後（例：「外見」の外枠を消した場合） 髪型: ショート: "short hair" 服装: シャツ: "shirt"

現在選択しているファイルの一番外側の階層（ルート）に、まったく新しい空のカテゴリーを追加します。 全く新しいジャンルを作り始めたい時の起点として使用します。

技術仕様 1. 3層アーキテクチャ（設計）

本システムは、Stable Diffusion / Gradioという制約の多いWebアプリケーション環境下で安定して動作させるため、明確に役割が分担された3つのレイヤーで設計されています。

Layer 1: Gradio (Python) ── 永続化とファイルシステム

ファイルの読み書き（YAMLセーブ、Base64デコード、画像PNG書き出し、YAML読み込み）のみを担当。複雑なUI制御は一切行わず、命令を待ち受ける「入出力の港（ハブ）」として機能します。

Layer 2: EasyPromptSelector (JavaScript) ── フロント表示と状態監視

WebUI側のボタン生成、プロンプトへのタグ挿入、可視化レイヤー、マクロ置換、および入力欄イベント監視を担当。

Layer 3: 匠エディタ (JavaScript) ── 匠編集エディタロジック

YAMLパース、編集差分の差分計算、重複防止判定、ドラッグ＆ドロップされた画像の超解像度圧縮、及び差分キャッシュ of 投函を担当。

技術仕様 2. YAML & 禁則仕様

本家Gradioのtextareaコンポーネントは、連続する空行（改行のみの行）があると、勝手にデータを詰めて削除してしまうという厄介な仕様があります。 EPS-Kaiは、YAMLを出力する際に独自の改行処理を挟んでこれを回避しています。

// 改行のみの行に「半スペース」を自動挿入して、Gradio側の自動削除バグを完全にすり抜ける return yamlStr.replace(/\n\n/g, '\n \n');

シングルクォート（ ' ）のエスケープ禁則処理

Python（Gradio）とフロントJSの間でプロンプトテキストを転送する際、文字列の中に シングルクォート（ ' ） がそのまま残っていると、JSパーサーが文字列表記の終端を誤認し、SyntaxError（Unterminated string literal）を引き起こして拡張機能全体の動作が永久停止（クラッシュ）する致命的なバグが懸念されます。

エスケープ禁則ルール：
プロンプトデータ内にシングルクォートが含まれている場合、描画処理やDOM組み立ての直前において、正規表現を用いてシングルクォートを確実に \' へと置換（エスケープ）します。これにより「バリア構造」といった大げさなDOM遮断を張ることなく、最も軽快かつスマートに構文破壊を100%根底から防御します。

技術仕様 3. 双方向通信システム (手紙と郵便受けモデル)

完全に独立して動いている「EPS-Kai（本家）」と「匠エディタ」、および「Gradio（Python側）」は、ブラウザの `localStorage` と非同期フラグ通知によって、シームレスに同期しています。

1. 匠エディタ側での保存

画像を保存すると、エディタはキャッシュデータベース of 自分のキーを削除し、さらに郵便受けである `_gg_updates` 配列に更新されたキーを放り込みます。

2. 本家EPS-Kai側の回収

本家側のチェッカーが `_gg_updates`（匠からの手紙）を発見すると、差分があったアイテムのみを通信スキャンし、一瞬でボタンの `◇` を `◉` へとリペイントします。

3. 自動回収と消去

スキャンが成功すると、郵便受け（`_gg_updates`）は自動的に破棄され、次のセーブを待ち受けます。

技術仕様 4. Selected_Index による「日本語ボタン」の完全復元

通常、WebUIで生成された画像には、どのプロンプト文字列が使用されたかという「PNG Info」のみが記録されます。 しかし、これだけでは再度画像をWebUIに投げ戻した際に、「どのカテゴリの、どのボタンを押してそれらを入力したのか」という情報が欠落しており、ボタンのハイライト状態が一切復元できません。

EPS-Kaiは、生成されるPNG画像データのPNG chunk内に、独自のキーである Selected_Index データをシリアライズして記録します。

PNGメタデータに書き出されるSelected_Indexの構造

Selected_Index: [
  { "tab": 0, "category": "hair-style", "index": 12 },
  { "tab": 2, "category": "outfit-uniform", "index": 5 }
]

PNGドロップ時の完全アクティブ復帰プロセス

【最大の魔法：日本語ボタンがそのまま残る！】
ユーザーが画像をWebUIの「PNG Info」にドラッグ＆ドロップした際、EPS-Kaiの自動復元リスナーが起動します。

もしプロンプトの英語テキスト（例：1girl, blue hair, school uniform）だけを頼りに復元しようとすると、どれがどのボタン由来なのか判別不可能です。
しかし、この Selected_Index には「タブ」「カテゴリ」「インデックス」という物理的なボタン位置が記録されています。この情報を使って直接アクティブ化するため、画像を読み込んだ瞬間、あなたが作った『日本語のボタン』たちが、当時押されていた時と寸分の狂いもなく、100%そのまま同じ状態で美しくピカッと光って復元されます！

技術仕様 5. YAMLデータの構築・多層構造ルールとRM展開

EPS-Kaiの圧倒的な表現力は、ボタンを定義する YAML ファイルの記述規則にあります。用途に合わせていくつかの柔軟なフォーマット、および多重ネストによる階層化をサポートしています。

① ドノーマル（単純な登録）

ボタン名とプロンプト値を「名前 ＋ コロン(:) ＋ ダブルクォート("")で囲ったプロンプト」の順で記述します。右辺値は常にダブルクォートで保護することを強く推奨します。

エメラルド髪: "emerald hair"
正面: "straight"

② CSOEフラグの付与（タグプレフィックス）

画像スキャン判定用のバッジフラグとして機能させたいボタンは、必ず「名前 [フラグ]（半角スペース1個を空けて[C]などのフラグ）」の順で記述します。これにより、UI上にカテゴリー絵文字アイコンが自動合成されます。

エメラルド髪 [C]: "emerald hair"

③ パイプ記号「|」による色コード（カラーボックス）連携

プロンプト値の後ろに | #カラーコード を付与することで、ボタン内に 14px 四方のカラーボックスをダイレクトにインライン描画させることができます。

エメラルド髪: "emerald hair | #50c878"

④ 複合的な構築（フラグ ＋ カラーコード）とプレビュー404エラーの完全回避

アイコンとカラーボックスの両方を備えた極めて高機能なボタンを記述する際の完全なフォーマット規則です。

エメラルド髪 [C]: "emerald hair | #50c878"
💡 サムネ消失を防ぐ『純粋スプリット化』処理のキモ：
ホバープレビュー（サムネイル表示）が起動した際、単に |# を検出して画像候補から排除してしまうと、フォルダに実在する専用画像（エメラルド髪 [C].png）があってもプレビューが消失するバグが生じます。
これを防ぐため、プレビュー画像探索ロジックは、値から色コード部分を一時的にスプリットして排除した純粋なプロンプト名（emerald hair）だけを取り出し、ボタン名と安全にマッチングさせます。これにより無駄な404エラーを100%完璧に遮断します。

⑤ YAMLインデントによる「多層構造（入れ子）」とアコーディオン展開

半角スペース2つ（または4つ）による階層的なインデントを挟むことで、ボタンを多層フォルダ化（ネスト）できます。以下は「表情」をテーマにした完全な設計例です。 

表情:
  笑顔系:
    にっこり [C]: "smile"
    悪だくみ笑顔 [C]: "evil smile"
    照れ笑い [C]: "shy smile"
  悲しみ:
    ほろり涙 [C]: "crying"
    号泣 [C]: "sobbing"
  怒り:
    激怒 [C]: "angry"
    不機嫌ぷく顔 [C]: "pouting lips"

⑥ カテゴリ全体のボタン化（一括展開・RMランダム抽選）

「表情」や「笑顔系」といったカテゴリ名（親ジャンル）も1つの独立したボタンとして表示されます。 このボタンを利用して、以下のようなマクロパスキーを構築できます。 

%%@表情@%%           # 表情カテゴリ内のすべてのプロンプトからランダムで1つを自動抽選！
%%@表情:笑顔系@%%       # 多層ネストされた「笑顔系」の配下のみを対象として自動抽選！

技術仕様 6. 拡張性：マスターマップでの独自フラグ追加

プログラムコードのいたる所に特定の判定記号を直接書き込んでしまう（ハードコーディング）状態を完全に廃止しました。 すべては、コードの先頭に定義された **`IMAGE_BADGE_MAP`** および **`TAG_PREFIX_MAP`** を参照して動作します。

【実践】新しいフラグ [W] 武器 (Weapon) を追加する場合
EPS-KaiのメインJSファイル冒頭にある2つの配列に、以下の1行ずつを書き足すだけです。

const TAG_PREFIX_MAP = { '[C]': { icon: '👤', label: 'Character' }, '[S]': { icon: '❤️', label: 'Series/Style' }, '[O]': { icon: '👕', label: 'Outfit' }, '[E]': { icon: '💎', label: 'Effect/Accessory' }, '[W]': { icon: '⚔️', label: 'Weapon' } // ← これを追加！ }; const IMAGE_BADGE_MAP = { 'C': { label: '◉ C', color: '#3b82f6' }, 'S': { label: '◉ S', color: '#10b981' }, 'O': { label: '◉ O', color: '#f59e0b' }, 'E': { label: '◉ E', color: '#8b5cf6' }, 'W': { label: '◉ W', color: '#94a3b8' }, // ← これを追加！ 'G': { label: '★ G', color: '#facc15' } };

これだけで完了です！ ロジックコードには1文字も触れることなく、本家EPS-Kaiのボタン表示、匠エディタのフラグトグル、画像名保存、ホバー時のバッジなど、システム全体が [W] フットバッジに全自動で対応・連動し始めます。

⚠️ 【超重要：Gフラグに関する絶対禁則事項】
マスターマップ内の 'G' キー（汎用バッジ：label: '★ G'）は、専用画像が存在しない場合にフォールバック探索を行うためのシステム予約キーです。この 'G' フラグの名称変更・削除は、システムを致命的に破壊する恐れがあるため「変更不可・削除厳禁」として運用してください。

技術仕様 7. なぜこの構造なのか

Gradio環境でのWebUI拡張開発は、頻繁に画面が再生成されたり、DOM（HTML要素）が丸ごと削除されたりする、非常に不安定な環境です。
このシステムは、サーバーへの問い合わせを極限まで減らし、「ブラウザ側のローカルストレージ（キャッシュ）」と「非同期通知」を主軸に設計したことで、**驚異的な動作の軽さ、無駄のない画像同期、それからWebUIの再読み込みにもへこたれない強力な復旧力**を担保することができました。

「必要だからこそ削り、必要だからこそ徹底してスマートにした」 この引き算の思想こそが、EPS-Kaiおよび匠エディタを支えるコアロジックです。
